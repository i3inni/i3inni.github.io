#1 댓글 좋아요 상태 유지 & 카운트 중복 증가

🔍 문제 인식

게시글을 나갔다가 다시 들어오면 내가 좋아요한 댓글의 하트가 초기화되고, 좋아요를 누를 때마다 카운트가 1이 아닌 2씩 증가했다. 두 버그가 동시에 발생해 UX가 심각하게 훼손된 상태였다.

💡 해결방안 선택지

A. 댓글 조회 API 재호출로 서버 상태 최신화

좋아요 토글 후 매번 전체 댓글을 서버에서 다시 받아오는 방식

B. 백엔드에서 isLiked 포함 응답 + 프론트 낙관적 업데이트 ✅ 선택

서버가 isLiked 상태를 직접 응답하고, 프론트는 해당 댓글만 로컬 업데이트

✅ 선택 이유

A안은 좋아요 한 번에 전체 댓글을 재조회하므로 불필요한 DB 쿼리와 네트워크 비용이 발생한다. B안은 서버에서 isLiked를 응답에 포함시켜 초기화 문제를 근본 해결하고, 프론트는 해당 댓글만 로컬 업데이트(updateCommentLike())해서 서버 재호출 없이 즉각 반응하게 만든다. 또한 Lombok의 boolean 필드 setter 네이밍 문제(isLiked → setLiked())를 직접 필드 세팅 방식으로 우회하는 게 더 안정적이었다.

📈 얻은 효과

  • 좋아요 토글 시 댓글 전체 재조회 API 호출 제거 → 네트워크 요청 수 감소
  • 하트 상태가 페이지 재진입 후에도 정확히 유지됨
  • @PreUpdate 제거로 좋아요 시 "수정됨" 표시 버그도 동시 해결

📚 알게 된 개념

  • Lombok의 isXxx 필드에 대한 setter 네이밍 규칙 (setXxx() 생성, prefix 제거)
  • JPA @PreUpdate의 동작 범위 — 연관 엔티티 변경도 트리거할 수 있음
  • 낙관적 업데이트(Optimistic Update) 패턴의 적용 기준과 롤백 처리 필요성

💭 아쉬운 점

  • 낙관적 업데이트 후 서버 오류 시 롤백 로직을 구현하지 않아, 네트워크 실패 시 UI와 서버 상태가 불일치 상태로 남을 수 있음
  • isLiked 직접 필드 세팅 우회 대신 Lombok 설정을 정비해 정규 방식으로 처리했다면 더 명확했을 것

#2 게시글 목록 댓글 수 — 대댓글 미포함 오류

🔍 문제 인식

게시판 목록의 댓글 수와 실제 상세 페이지의 댓글 수가 달랐다. 목록에서는 대댓글을 제외한 루트 댓글만 카운트하고 있었으며, 삭제된 댓글도 카운트에 포함되는 문제가 있었다.

💡 해결방안 선택지

A. 게시글 조회 시 매번 댓글 수 서브쿼리 포함

각 게시글마다 COUNT 서브쿼리 → N+1 가능성

B. postIds 배치 쿼리로 한 번에 카운트 ✅ 선택

IN 절로 전체 게시글 ID를 한 번에 조회 후 Map으로 매핑

✅ 선택 이유

목록 API는 다수의 게시글을 한꺼번에 반환한다. A안처럼 게시글마다 서브쿼리를 실행하면 게시글 N개 → 쿼리 N+1번이라는 전형적인 N+1 문제가 발생한다. B안은 countNonDeletedCommentsByPostIds()를 배치 쿼리로 만들어 한 번의 IN 쿼리로 전체 결과를 가져오고 Map<Long, Long>으로 매핑하므로 쿼리 횟수가 1회로 고정된다.

📈 얻은 효과

  • 게시글 목록 API의 댓글 수 관련 쿼리가 N번 → 1번으로 감소
  • 전체/카테고리/후기 등 모든 목록 API에 일괄 적용해 일관성 확보
  • 대댓글 포함, 삭제 댓글 제외 카운트로 정확도 개선

📚 알게 된 개념

  • N+1 문제의 발생 원인과 배치 쿼리(IN절) + Map 매핑 패턴을 통한 해결 방법
  • Spring Data JPA에서 @Query로 집계 쿼리 작성 시 결과를 List<Object[]>로 받아 Map 변환하는 패턴

💭 아쉬운 점

  • 댓글 수를 DB 컬럼에 캐싱(역정규화)하지 않아 매 목록 조회 시 COUNT 쿼리가 실행 — 게시글이 많아질수록 부하 증가
  • IN절 쿼리는 ID 수가 늘어날수록 성능이 저하될 수 있어, 페이지네이션 단위 캐싱을 함께 고려했으면 더 좋았을 것
BUG

#3 지도 확대 버튼 단계별 동작 불량

🔍 문제 인식

+ 버튼이 첫 번째 클릭만 작동하고 이후엔 눌러도 지도가 확대되지 않았다. 원인은 onCameraIdle에서 latitudeDelta를 이용해 zoom 레벨을 역산할 때 Math.round() 오차로 인해 lastCameraRef가 잘못된 값으로 덮어써져 이후 zoom 변경이 감지되지 않는 것이었다.

💡 해결방안 선택지

A. Math.round 대신 소수점 허용 역산 보정

역산 공식을 정밀하게 수정해 오차 최소화

B. onCameraIdle에서 params.zoom 직접 사용 ✅ 선택

역산 자체를 제거하고 카메라 이벤트가 내려주는 zoom 값을 그대로 사용

✅ 선택 이유

역산 자체가 근본적으로 오차를 유발하는 구조이므로, 정밀도를 높여도 동일한 문제가 재발할 수 있다. onCameraIdle 타입에 zoom?: number를 추가해 SDK가 제공하는 정확한 zoom 값을 그대로 쓰는 게 가장 단순하고 안정적인 해결책이었다. + 버튼도 Math.min(18, currentZoom + 1)로 단순화해 최대 줌 상한까지 처리했다.

📈 얻은 효과

  • 확대 버튼이 클릭할 때마다 정확히 1단계씩 확대됨
  • 불필요한 역산 로직 제거로 코드 단순화
  • 최대 줌(18) 상한 처리로 범위 초과 오류 방지

📚 알게 된 개념

  • 지도 SDK의 latitudeDelta와 zoom 레벨 간 변환 공식의 부정확성
  • 부동소수점 오차(Math.round)가 상태 비교에 미치는 영향
  • SDK 이벤트 콜백이 제공하는 값을 신뢰하는 것이 역산보다 안전하다는 원칙

💭 아쉬운 점

  • 최댓값(18) 상한만 처리하고 최솟값(min zoom) 하한 처리를 추가하지 않아 과도한 축소가 가능한 상태
  • SDK 타입을 임시로 확장해 해결했는데, SDK 버전 업데이트 시 타입 정의 충돌 가능성이 있음
FEAT

#4 축제 목록에서 선택 시 지도 포커스 이동

🔍 문제 인식

축제 목록에서 항목을 선택해도 지도가 해당 위치로 이동하지 않아 사용자가 직접 지도를 드래그해야 했다. 지도와 목록이 연동되지 않아 UX가 단절되는 문제였다.

💡 해결방안 선택지

A. 선택 시 지도 탭으로 화면 전환 후 좌표 전달

화면 이동이 발생해 현재 목록 컨텍스트를 잃음

B. mapRef로 지도 직접 제어 — animateCameraTo() ✅ 선택

화면 전환 없이 지도만 해당 좌표로 부드럽게 이동

✅ 선택 이유

목록과 지도가 같은 화면에 공존하는 구조이므로 화면 전환 없이 mapRef.current.animateCameraTo()로 지도만 부드럽게 이동시키는 것이 가장 자연스럽다. zoom 15로 고정해 축제 위치가 명확하게 보이도록 했다.

📈 얻은 효과

  • 목록 선택 → 지도 자동 이동으로 UX 연속성 확보
  • 사용자가 직접 지도를 탐색하는 단계 제거

📚 알게 된 개념

  • useRef를 통한 지도 인스턴스 직접 제어 패턴
  • animateCameraTo()의 애니메이션 카메라 이동과 즉시 이동(moveCamera())의 차이

💭 아쉬운 점

  • 포커스 이동 시 zoom을 15로 고정해, 사용자가 이전에 설정한 zoom 레벨이 무시됨 — 기존 zoom 상태를 유지하는 방식이 더 자연스러웠을 것

#5 지도 사이드메뉴 — 뒤로가기 시 탭·스택 꼬임

🔍 문제 인식

지도 사이드메뉴에서 "저장한 축제", "앱 설정" 등을 누르면 마이페이지 탭으로 이동하는데, 뒤로가기를 해도 마이페이지에 그대로 머물고 지도 탭으로 돌아오지 않았다. 이후 마이페이지 탭을 다시 누르면 이전에 진입했던 화면이 스택에 남아 중첩되는 문제도 있었다.

💡 해결방안 선택지

A. StackActions.popToTop() + 탭 전환

popToTop 실행 시 일부 환경에서 에러 발생

B. from 파라미터 감지 → router.replace로 마이페이지 루트 초기화 후 지도 탭 전환 ✅ 선택

스택을 replace로 초기화해 잔류 없이 지도 탭으로 전환

✅ 선택 이유

popToTop은 Expo Router 환경에서 에러가 발생하는 케이스가 있었다. 대신 각 화면에 from: 'map' 파라미터를 전달해 진입 경로를 식별하고, router.replace로 마이페이지 스택을 루트로 초기화한 후 지도 탭으로 전환하면 스택 잔류 없이 깔끔하게 원래 자리로 돌아올 수 있었다.

📈 얻은 효과

  • 사이드메뉴 → 화면 → 뒤로가기 시 지도 탭으로 정확히 복귀
  • 마이페이지 탭 재진입 시 스택 중첩 문제 완전 제거
  • 미사용 import 제거로 코드 정리

📚 알게 된 개념

  • Expo Router의 탭 기반 네비게이션과 스택 네비게이션의 상호작용 구조
  • router.push vs router.replace의 스택 처리 차이
  • 진입 경로 파라미터(from)를 활용한 조건부 뒤로가기 처리 패턴

💭 아쉬운 점

  • from 파라미터를 URL에 노출하는 방식이라, URL 직접 조작 시 잘못된 값이 들어올 수 있음 — 전역 상태(Context)로 관리했다면 더 견고했을 것

#6 좋아요 하트 애니메이션 & 낙관적 업데이트

🔍 문제 인식

게시글·댓글 좋아요를 누를 때 서버 응답을 기다린 후 refetchPost()를 호출해서 화면이 깜빡이고 응답 지연이 체감됐다. 인터랙션 피드백도 없어 눌렸는지 확인하기 어려웠다.

💡 해결방안 선택지

A. 서버 응답 후 전체 데이터 재조회 유지

항상 서버 기준 최신 상태이나, 응답 전까지 화면 지연

B. localLikeCount 낙관적 업데이트 + Spring 애니메이션 ✅ 선택

클릭 즉시 로컬 상태 업데이트 + 하트 scale 애니메이션

✅ 선택 이유

좋아요는 실패 가능성이 낮고 롤백이 필요한 경우가 드문 작업이다. 낙관적 업데이트로 클릭 즉시 UI를 반영하고, Spring 애니메이션으로 하트가 튀는 피드백을 주면 서버 지연과 무관하게 사용자가 즉각적인 반응을 느낄 수 있다. 게시글 전체 리패치는 불필요한 렌더링을 유발하므로 제거했다.

📈 얻은 효과

  • 좋아요 클릭 즉시 반응 — 서버 응답 대기 시간 체감 제거
  • 전체 게시글 리패치 제거로 불필요한 렌더링 차단
  • 하트 scale 애니메이션으로 인터랙션 피드백 제공

📚 알게 된 개념

  • 낙관적 업데이트(Optimistic Update)의 적용 기준 — 실패 가능성이 낮고 즉각 피드백이 중요한 인터랙션에 적합
  • React Native Reanimated의 spring 애니메이션과 withSequence를 활용한 scale 효과

💭 아쉬운 점

  • 낙관적 업데이트 후 서버 오류 시 롤백 로직이 없어, 요청 실패 시 카운트가 틀린 값으로 표시될 수 있음
  • 연속으로 빠르게 좋아요를 토글하면 애니메이션이 중첩되어 어색해 보이는 문제가 남아 있음
FEAT

#7 채팅 날짜 구분선 동적화

🔍 문제 인식

채팅 내역의 날짜 구분선이 "오늘"로 하드코딩되어 있어 어제 또는 더 이전 메시지에도 "오늘"이 표시됐다. 실제 메시지 시간 기반의 구분선이 필요했다.

✅ 해결 및 선택 이유

Message 타입에 createdAt 필드를 추가하고, 메시지 목록을 렌더링할 때 이전 메시지와 날짜가 다르면 구분선을 삽입하는 방식으로 동적화했다. 오늘/어제는 상대적 표현으로, 그 이전은 "N년 N월 N일" 형식으로 출력해 가독성을 높였다.

📈 얻은 효과

  • 날짜 구분선이 실제 메시지 날짜 기준으로 정확하게 표시
  • 오랜 채팅 히스토리도 날짜별로 구분되어 가독성 향상

📚 알게 된 개념

  • 리스트 렌더링 시 이전 아이템과 비교해 구분선을 조건부 삽입하는 패턴
  • 날짜 포맷팅 — 오늘/어제 판단 로직과 절대 날짜 표기 분기 처리

💭 아쉬운 점

  • 날짜 비교 로직이 클라이언트 로컬 시간 기준이라, 서버와 타임존이 다른 환경에서 날짜 경계에 구분선이 잘못 표시될 수 있음

#8 행사 상세 조회수 중복 카운트 제거

🔍 문제 인식

행사 상세 페이지 진입 시 조회수가 2씩 증가했다. useEventDetail 훅과 EventDetailHeader 컴포넌트 두 곳에서 각각 logClick()을 호출하고 있었다.

✅ 해결 및 선택 이유

EventDetailHeader에서 이미 호출 중이므로 useEventDetail 훅의 logClick 호출을 주석 처리해 단일 호출로 정리했다. 호출 책임을 UI 컴포넌트(Header)에 두는 것이 데이터 훅과의 관심사 분리 원칙에도 부합한다.

📚 알게 된 개념

  • 훅과 컴포넌트 간 사이드이펙트 호출의 중복 방지 — 관심사 분리 원칙 적용
  • 중간 투입 시 코드 중복·사이드이펙트 탐지 방법

💭 아쉬운 점

  • 해결 방식이 단순 주석 처리라 이후 코드 수정 시 같은 중복 문제가 재발할 수 있음 — 훅 내부에서 중복 호출을 방어하는 로직이 있었다면 더 견고했을 것

#9 채팅 기능 개선 — 나가기 · 미읽음 · 이름 표시

🔍 문제 인식

채팅방을 나가도 재진입 시 같은 방이 재사용되어 "새 채팅방 생성" 흐름이 깨졌고, 나간 상대와 채팅 중인 경우 입력창이 그대로 노출됐다. 미읽음 카운트에 내가 보낸 메시지도 포함되는 문제, 나간 상대의 이름이 비어있는 문제도 있었다.

💡 해결방안 선택지 (나가기 흐름)

A. 나가기 시 방 즉시 삭제

상대방도 히스토리를 잃게 됨

B. directRoomKey 초기화 + 둘 다 나갔을 때만 영구 삭제 ✅ 선택

한 명이 나가도 상대방 히스토리 유지, 둘 다 나가면 완전 삭제

✅ 선택 이유

나가기 시 directRoomKey를 null로 초기화하면, 이후 재채팅 시 기존 방을 찾지 못해 항상 새 방이 생성된다. 한 명만 나간 경우 상대방 히스토리를 보호하고, 둘 다 나간 경우에만 메시지·멤버·방을 영구 삭제해 DB 공간도 확보했다. 미읽음 카운트는 countByRoomAndCreatedAtAfterAndSenderIdNot 쿼리로 발신자를 제외해 정확도를 높였다.

📈 얻은 효과

  • 나가기 후 재채팅 시 항상 새 방 생성 — 이전 히스토리 오염 없음
  • 상대가 나간 경우 입력창 대신 안내 메시지 표시로 UX 명확화
  • 미읽음 카운트에서 내가 보낸 메시지 제외 — 정확한 미읽음 뱃지
  • 둘 다 나간 방 자동 정리로 불필요한 데이터 제거
  • 나간 상대 이름 "알 수 없음" 처리로 빈 값 노출 방지

📚 알게 된 개념

  • 채팅방 상태 관리 — directRoomKey를 활용한 방 중복 생성 방지 패턴
  • 소프트 삭제(leftAt 기록)와 하드 삭제(영구 제거) 기준 설계
  • Spring Data JPA 메서드 네이밍으로 복합 조건 쿼리 생성 (countByRoomAndCreatedAtAfterAndSenderIdNot)
  • 프론트에서 leftAt 필드로 상대방 퇴장 상태를 감지해 UI를 조건부 렌더링하는 패턴

💭 아쉬운 점

  • directRoomKey를 null로 초기화하는 방식이라 DB에 null 값이 남아, 이후 데이터 정합성 확인이 까다로울 수 있음
  • 둘 다 나갔을 때 메시지·멤버·방을 일괄 삭제하는 로직이 단일 트랜잭션으로 묶여 있지 않아, 중간 실패 시 부분 삭제 상태가 남을 가능성이 있음
