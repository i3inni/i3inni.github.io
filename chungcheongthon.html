#1 JWT 인증 시스템

🔍 문제 인식

해커톤 특성상 빠르게 인증 시스템을 구축해야 했다. 세션 방식은 서버에 상태를 저장해야 해서 수평 확장이 어렵고, 별도 세션 저장소 설정이 필요해 해커톤 일정에 맞지 않았다.

💡 해결방안 선택지

A. 세션 기반 인증

서버에 세션 저장, 별도 저장소 필요, 수평 확장 시 세션 공유 문제

B. JWT + Spring Security ✅ 선택

Stateless, 서버 상태 없음, 토큰 자체에 정보 포함, 빠른 구현 가능

✅ 선택 이유

JWT는 Stateless 구조라 별도 저장소 없이 토큰 하나로 인증이 완결된다. JJWT 라이브러리로 발급·검증 로직을 빠르게 구현할 수 있고, @AuthenticationPrincipal로 컨트롤러에서 사용자 정보를 바로 주입받아 해커톤 제한 시간 내에 안정적인 인증 레이어를 완성할 수 있었다.

📈 얻은 효과

  • 서버 재시작 후에도 토큰 유효 — 세션 유실 문제 없음
  • 프론트엔드와 Authorization 헤더 하나로 인증 연동 완료
  • 토큰 유효기간(24h) 설정으로 보안과 편의성 균형 유지

📚 알게 된 개념

  • JWT 구조 (Header.Payload.Signature)와 서명 검증 원리
  • Spring Security Filter Chain에서 JWT 필터를 삽입하는 위치와 순서
  • UsernamePasswordAuthenticationFilter 앞에 커스텀 필터를 두는 이유

💭 아쉬운 점

  • Refresh Token을 구현하지 않아 Access Token(24h) 만료 시 자동 갱신이 불가 — 사용자가 매일 재로그인해야 함
  • 로그아웃 후에도 발급된 토큰이 만료 전까지 유효한 상태로 남아 강제 무효화 불가 (블랙리스트 미구현)

#2 일정 CRUD API 설계

🔍 문제 인식

일정 앱의 핵심은 다양한 조회 패턴이다. 전체 조회뿐 아니라 특정 날짜, 날짜 범위별 조회가 필요했고, 완료 처리와 우선순위 변경을 각각 독립적인 API로 제공해야 프론트엔드가 필요한 것만 호출할 수 있었다.

💡 해결방안 선택지

A. 단일 PUT으로 전체 필드 업데이트

완료, 우선순위, 내용 변경을 하나의 엔드포인트로 처리 → 프론트가 불필요한 필드까지 전송

B. 기능별 PATCH 엔드포인트 분리 ✅ 선택

완료 처리, 우선순위 변경을 각각 독립 PATCH로 분리 → 최소 데이터 전송, 명확한 의도

✅ 선택 이유

완료 처리와 우선순위 변경은 각각 독립적인 사용자 행동이다. PATCH로 분리하면 프론트엔드가 변경할 필드만 전송하고, 서버도 해당 필드만 업데이트하면 되어 불필요한 데이터 전송과 검증을 줄일 수 있다. 조회도 날짜별·범위별로 분리해 캘린더 뷰와 리스트 뷰가 각각 최적화된 API를 사용할 수 있게 했다.

📈 얻은 효과

  • 기능별 엔드포인트 분리로 프론트엔드가 필요한 것만 호출 가능
  • 날짜/범위 조회 분리로 캘린더·리스트 뷰 각각 최적 응답 제공
  • REST 원칙에 맞는 API 설계로 가독성·유지보수성 향상

📚 알게 된 개념

  • PUT(전체 교체)과 PATCH(부분 수정)의 차이와 올바른 사용 기준
  • JPA @Query로 날짜 범위 조건 쿼리 작성 방법

💭 아쉬운 점

  • 소프트 삭제(deletedAt 플래그) 없이 하드 삭제만 구현해 실수로 삭제한 일정 복구가 불가능함
  • 동시에 두 사용자가 같은 일정을 수정할 경우 충돌 처리 로직(낙관적 락)이 없어 나중 요청이 이전 값을 덮어씀
FEAT

#3 AI 일정 분석 — Gemini API 연동

🔍 문제 인식

단순 일정 저장에서 끝나지 않고 사용자에게 의미 있는 인사이트를 제공해야 했다. 직접 분석 로직을 짜는 것은 해커톤 시간 내에 불가능했고, 외부 AI API 장애 시 서비스 전체가 멈추는 상황도 막아야 했다.

💡 해결방안 선택지

A. OpenAI GPT-4 연동

성능은 우수하나 비용이 높고, 해커톤 API 키 발급 시간 소요

B. Google Gemini 2.0 Flash + 폴백 처리 ✅ 선택

무료 티어 제공, 빠른 발급, API 실패 시 통계 기반 폴백으로 안정성 확보

✅ 선택 이유

Gemini는 무료 티어가 넉넉하고 API 키 발급이 즉시 가능해 해커톤에 적합했다. 핵심은 폴백 처리였다 — AI API가 실패하더라도 완료율, 미완료 수 등 DB에서 직접 집계한 통계 정보를 응답으로 내려주도록 설계해 외부 서비스 장애가 전체 서비스 장애로 이어지지 않게 했다.

📈 얻은 효과

  • AI API 장애 시에도 통계 기반 응답으로 서비스 연속성 보장
  • 사용자 일정 데이터를 기반으로 3가지 실행 가능한 인사이트 제공
  • 외부 의존성을 인터페이스로 추상화해 모델 교체가 용이한 구조 설계

📚 알게 된 개념

  • 외부 API 연동 시 폴백(Fallback) 처리의 중요성과 구현 패턴
  • 서비스 인터페이스 분리(AiService / GeminiAiService)로 구현체 교체를 유연하게 만드는 방법

💭 아쉬운 점

  • 동일한 일정 데이터로 반복 요청 시 AI API를 매번 새로 호출 — 응답 캐싱이 없어 API 비용 낭비
  • 프롬프트가 코드에 하드코딩되어 있어 인사이트 품질 개선을 위해서는 코드 수정·재배포가 필요한 구조
FEAT

#4 Docker 배포 환경 구성

🔍 문제 인식

팀원 각자의 로컬 환경이 달라 "내 PC에서는 되는데 서버에서 안 된다"는 문제가 생길 수 있었다. 또한 개발 환경(로컬 DB)과 배포 환경(외부 DB) 설정을 코드 변경 없이 분리해야 했다.

💡 해결방안 선택지

A. 서버에 직접 Java 설치 후 jar 실행

환경 의존성이 서버 Java 버전에 묶임, 환경 재현 어려움

B. Dockerfile + 프로파일 분리 ✅ 선택

컨테이너로 환경 고정, application-docker.yml로 배포 설정 분리

✅ 선택 이유

OpenJDK 21 기반 Dockerfile로 실행 환경을 이미지에 고정하면 서버 Java 버전과 무관하게 동일한 환경이 보장된다. application-docker.yml을 별도로 구성해 Docker 실행 시 자동으로 배포용 설정이 적용되도록 해 코드 수정 없이 환경을 전환할 수 있었다.

📈 얻은 효과

  • 개발·배포 환경 설정 분리로 코드 수정 없이 환경 전환
  • 팀원 간 환경 차이로 인한 "내 PC에서만 됨" 문제 제거
  • 이미지 빌드 한 번으로 어디서든 동일하게 실행 가능

📚 알게 된 개념

  • Dockerfile 멀티스테이지 빌드 없이 단순 실행 이미지 구성하는 방법
  • Spring Boot 프로파일(spring.profiles.active)을 Docker 환경 변수로 제어하는 패턴

💭 아쉬운 점

  • CI/CD 파이프라인을 구성하지 않아 코드 변경마다 수동으로 이미지를 빌드하고 서버에 배포해야 하는 번거로움
  • 멀티스테이지 빌드를 적용하지 않아 최종 이미지에 Gradle 빌드 도구까지 포함되어 이미지 크기가 불필요하게 큼
