프로젝트 개요

RunBoo는 런닝을 단순한 운동이 아닌 게임처럼 즐길 수 있도록 설계한 런닝 앱입니다. 5km·10km 기록을 기반으로 티어를 부여하고, 도전과제 클리어를 통해 뱃지를 획득하는 게임화(Gamification) 요소를 통해 런닝 지속성을 높이는 것이 핵심 목표입니다. 고스트 모드, 주변 러너 탐색, AI 분석 리포트 등의 기능도 제공합니다.

FEAT

#1 티어 시스템 — 백엔드

🔍 문제 인식

5km·10km 기록 기반으로 티어를 부여해야 하는데, 두 거리 타입의 티어를 독립적으로 관리해야 했다. 같은 사용자가 거리 타입별로 처음 기록하면 INSERT, 이후 갱신하면 UPDATE가 필요한데 이를 클라이언트에서 분기하면 복잡해지고 오류 가능성이 높았다.

💡 해결방안 선택지

A. 클라이언트가 INSERT/UPDATE를 구분해 다른 API 호출

클라이언트 복잡도 증가, 상태 판단 오류 가능성

B. 서버 단일 API에서 Upsert 처리 ✅ 선택

서버가 기존 기록 여부를 판단해 UPDATE/INSERT 자동 분기

✅ 선택 이유

클라이언트는 단순히 "티어 저장" 요청만 보내고, 서버가 findByUserAndDistanceType()으로 기존 기록을 조회해 있으면 updateTier(), 없으면 신규 저장하도록 구현했다. @Transactional + JPA Dirty Checking 덕분에 UPDATE 시 별도 save() 호출 없이 트랜잭션 커밋 시점에 변경이 자동 반영된다.

📈 얻은 효과

  • 클라이언트는 항상 단일 엔드포인트만 호출 — 분기 로직 불필요
  • 5km·10km 티어 독립 관리로 각 거리별 정확한 등급 표시
  • @JsonProperty("5k")로 프론트 값 형식과 정확히 매핑

📚 알게 된 개념

  • JPA Dirty Checking — 트랜잭션 내 엔티티 필드 변경 시 자동 UPDATE 발생 원리
  • @Enumerated(EnumType.STRING)과 @JsonProperty를 조합해 DB 저장 값과 JSON 응답 값을 각각 제어하는 방법

💭 아쉬운 점

  • 티어 변경 이력을 별도 테이블에 기록하지 않아 "언제 Bronze에서 Silver로 올랐는가" 같은 히스토리 추적이 불가
  • 배치 처리 없이 기록마다 즉시 티어를 업데이트하므로, 다수 사용자가 동시에 기록을 제출하면 DB 쓰기 부하 집중 가능성 있음
FEAT

#2 도전과제 시스템 — 백엔드

🔍 문제 인식

게임처럼 레벨이 순차적으로 잠금 해제되는 도전과제 시스템이 필요했다. 레벨을 한꺼번에 생성하면 DB 공간 낭비가 크고, 완료 시 뱃지 지급·알림·다음 레벨 오픈이 각각 별도 API로 나뉘면 프론트에서 여러 번 요청해야 하는 문제가 있었다.

💡 해결방안 선택지

A. 모든 레벨(1~30) 처음부터 생성 후 상태로 제어

DB에 30개 레코드 선생성, 불필요한 공간 낭비

B. 필요 시점에 지연 생성 + 상태 머신 + 단일 PATCH로 연쇄 처리 ✅ 선택

초기 1~3레벨만 생성, 완료 시점에 미래 레벨 추가 생성

✅ 선택 이유

초기화 시 1~3레벨만 생성하고 완료 시점에 ensureFutureLevelExists(level+3)으로 다다음 레벨을 LOCKED 상태로 추가 생성한다. 30레벨 상한도 처리했다. PATCH /progress 한 번으로 진행도 업데이트 → 완료 처리 → 뱃지 지급 → 레벨 오픈 → 알림까지 한 트랜잭션에서 연쇄 처리해 프론트 추가 요청을 없앴다. Math.min(progress + value, target)으로 진행도 오버플로도 방지했다.

📈 얻은 효과

  • 레벨 지연 생성으로 DB 레코드 최소화
  • 단일 PATCH로 완료·뱃지·레벨오픈·알림 연쇄 처리 → 프론트 추가 요청 0건
  • 멱등성 보장 — 중복 초기화 호출 시 early return으로 데이터 무결성 유지
  • Join Fetch로 완료 목록 조회 시 N+1 문제 방지

📚 알게 된 개념

  • 상태 머신 패턴(LOCKED → IN_PROGRESS → COMPLETED)으로 엔티티 라이프사이클 제어하는 방법
  • 멱등성(Idempotency) 설계 — 같은 요청을 여러 번 해도 결과가 동일하도록 보장하는 원칙
  • Join Fetch vs Lazy Loading의 차이와 N+1 문제 해결 방법

💭 아쉬운 점

  • 도전과제 유형(거리, 횟수 등)이 DB가 아닌 코드에 하드코딩되어 있어, 신규 유형 추가 시 재배포가 필요
  • 진행도·뱃지 지급·레벨 오픈이 하나의 트랜잭션에 묶여 있어 중간 단계 실패 시 전체가 롤백 — 부분 성공 처리 불가
FEAT

#3 프론트엔드 연동 — React + TypeScript

🔍 문제 인식

티어·도전과제 화면은 라이트/다크 모드를 모두 지원해야 했고, 컴포넌트마다 스타일이 흩어지면 유지보수가 어려워지는 문제가 있었다.

💡 해결방안 선택지

A. 컴포넌트 내부에 스타일 인라인 작성

빠르지만 색상·간격 값이 각 파일에 분산되어 일관성 깨짐

B. 중앙 theme.ts + getStyles(colorScheme) 패턴 ✅ 선택

색상·간격을 theme.ts 한 곳에서 관리, 컴포넌트는 getStyles()로 조회

✅ 선택 이유

theme.ts를 단일 진실 공급원(SSOT)으로 두고, 각 컴포넌트는 getStyles(colorScheme)를 호출해 테마에 맞는 스타일을 받아 쓴다. useColorScheme() 훅으로 시스템 다크모드를 자동 감지하므로 별도 상태 관리 없이 라이트/다크 전환이 이뤄진다.

📈 얻은 효과

  • 색상·간격 변경 시 theme.ts 한 곳만 수정하면 전체 반영
  • 라이트/다크 모드 자동 전환 — 별도 토글 로직 불필요
  • [Component].tsx + [Component].styles.ts 분리로 UI 로직과 스타일 관심사 분리

📚 알게 된 개념

  • 단일 진실 공급원(SSOT) 패턴 — 상수를 한 곳에서 관리해 일관성 보장
  • React Native useColorScheme() 훅의 동작 방식과 StyleSheet 동적 생성 패턴

💭 아쉬운 점

  • theme.ts에 색상 값이 리터럴로 하드코딩되어 있어, 브랜드 색상 변경 시 여러 상수를 일일이 수정해야 함
  • TypeScript 타입 정의 일부를 any로 처리한 부분이 남아 있어 컴파일 타임 오류 감지가 불완전함
